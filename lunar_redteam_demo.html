Lunar lander descent-design review

Check a descent and landing design against a structured record of past lunar-lander guidance, navigation and control failures.

Review a design

Paste a descent/landing design or concept of operations, or load an example. The review returns the historical failure modes the design appears exposed to, each with the missions where it occurred and a suggested mitigation.

LLM modes read the design for meaning instead of matching keywords. Agent mode runs three lightweight passes (extract facts → draft findings → audit false positives/false mitigations) and is the recommended mode for real ConOps. For best security, run python3 redteam.py --serve --provider parley and use Local server; the key stays in the Python process. Direct browser provider calls are still available, but require pasting a key into this page.

Failure record

Modern robotic soft-landing attempts, each normalized to a common schema: flight phase, subsystem, the proximate cause, the generalizable weakness, a sourced quote, and a confidence level. This is the record the review matches against.

How it works

There are two parts: the failure record and the review engine. The record stores what went wrong before; the engine compares a new ConOps against those reusable failure patterns.

Review pipelineTraceable evidence → structured JSON → review cards
01Source evidence

Public mishap reports, agency releases, and post-flight statements are reduced to traceable records.

02Failure memory

Each mission becomes a normalized schema with phase, subsystem, proximate cause, weakness, quote, source, and confidence.

03Design checks

Generalized failure modes ask diagnostic questions and link to precedent missions plus mitigations.

04Matcher / agent

Keyword mode matches terms. Agent mode extracts facts, drafts findings, and audits false positives.

05Structured findings

The model returns JSON: status, evidence, rationale, and confidence. The UI renders the cards.

Stored schemas

Mission failure record{ flight_phase, subsystem, sensor_or_actuator, proximate_cause, design_smell, mitigation_pattern, key_quote, source_url, confidence }
Reusable design check{ id, name, question, severity, present_signals[], mitigation_signals[], precedents[], recommendation }
LLM / agent finding{ check_id, status: flag|partial|pass, evidence, rationale, confidence }

Review modes

Demo (offline keyword). Transparent and deterministic, but literal: it matches words rather than meaning, so it can miss paraphrases or confuse a sensor mention with a mitigation.
LLM single pass. Sends the design text and check list to a model, which returns structured JSON findings in one call.
Agent (recommended). Runs three passes: extract supported design facts, draft findings against every failure mode, then audit for false positives/false mitigations. This is why the UI can render confidence, severity, evidence, precedents, and mitigations as separate elements.

Recommendations favor redundancy, dissimilar sensing, and graceful degradation over relying on a single string and making it more reliable.

Limitations

The record is small, so frequencies are reported as counts, not probabilities. Some causes rest on agency or company statements rather than full mishap reports; those carry a lower confidence level. The Demo matcher is a transparent placeholder for the LLM/Agent modes, not a substitute for them.
